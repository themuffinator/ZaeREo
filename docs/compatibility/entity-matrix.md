# Entity compatibility matrix

- Evidence: all 20 supplied BSP entity lumps plus the supplied Zaero spawn/item
  tables
- Audit date: 2026-07-13
- Runtime implementation: **in progress; see per-row evidence**

The BSPs contain 14,933 entity records and 132 distinct exact classnames. The
Rerelease baseline recognizes 105 of those names; the first table locks the 27
map-facing names that require Zaero implementations. Counts below are literal
entity-lump occurrences after pak0 < pak1 < pak2 resolution. They do not prove
that an entity currently spawns in ZaeREo.

The partial port currently resolves all 132/132 exact shipped names. This
registry closure does not weaken
the invariant below: a production package must resolve all 132 without a
placeholder or silent free unless the exact entity's compatibility contract is
an approved no-op.

The checked-in `docs/audits/bsp-entities.*` intentionally records the pristine
upstream 105/132 baseline. Reproduce the worktree-only 132/132 progress snapshot
without overwriting that evidence:

~~~powershell
python tools/audit_bsp_entities.py --assets-root "$env:ZAERO_LEGACY_ROOT" --rerelease-root src --pak pak0.pak --pak pak1.pak --pak pak2.pak --json-output build/audits/current-registry.json --markdown-output build/audits/current-registry.md
~~~

The generated local report must show zero missing names. A committed
current-registry report is added only when the project defines its regeneration
and baseline-update policy; until then this count is reproducible progress
evidence, not a VERIFIED release proof.

Every implementation must retain its exact classname, accepted keys and flags,
stable save identity, callbacks, resources, and cleanup. The historical
monster_sentien spelling is part of the mapper ABI.

[D-044](decisions.md#d-044--global-sv_flymove-duplicate-plane-delta) is a
cross-entity physics disposition for the FALLFLOAT rows E-022/E-023/E-025/
E-031/E-047 and Hover row E-050. Its [identity-locked audit and seven float32
goldens](../audits/flymove.md) retain the unmodified shared Rerelease solver
instead of Zaero's one-line global exact-repeat dead-stop. This adds static
caller/collision evidence only; no entity row advances without its live
always-windowed collision, save, lifecycle, and map fixtures.

[D-045](decisions.md#d-045--zdmflags-and-deathmatch-item-injection) is the
cross-entity deathmatch activation contract for E-001–E-008. Its
[normalized source/BSP audit](../audits/dm-injection.md) locks the historical
Sonic/Sniper/Flare/IRED/A2K/Flares/EMP/Shield order and proves that any one
live member suppresses the entire automatic pass. Injected entities use the
same registered `SpawnItem`/`droptofloor`/`Touch_Item` and saved native item
state as authored placements; the injector adds no callback or serialized
field. All 20 supplied maps already contain at least one member. The retained
schema-valid Debug runs prove the listed injection outcomes, identities, order,
offsets, final origins, and native Toss/trigger/Touch_Item/IR state, including
the real-brush partial-success and one-member controls. They were produced by
the legacy v1 launcher, however, and are historical behavior evidence only:
they must be repeated through the v2 two-stage window-before-mod/map protocol
before they can count as current runtime safety evidence. No E-001–E-008 row
advances without pickup/respawn/drop, save, and multiplayer evidence.

## Required registry invariant

Production validation must reject a release when:

- any of the 132 shipped exact classnames is unknown;
- an incomplete entity silently substitutes a placeholder;
- a recognized classname ignores a Zaero-required key/flag without a recorded
  no-op decision;
- callback, move, resource, or state registration is absent from JSON saves; or
- the normalized audit no longer matches these counts without an intentional
  baseline decision.

The committed generated BSP audit supersedes hand-maintained counts. Any
generated change must be reviewed before this matrix is updated.
